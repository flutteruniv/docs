# KBOYのFlutter大学アプリ（サロンアプリ）開発

サロンアプリの開発に join する場合は、下記をお読みください。

## 開発方針

フラットな組織で同じ方向性をもってプロジェクトを進める上では、一定の「規律」が必要です。  
ここで言う規律とはメンバーの「ロール」を決めて、ロールに応じた「ルール」を定めることです。  
メンバーの入れ替わりも激しいと想定して、できる限り個人に依存することは避け、チームを固定しない開発方針としています。

<img width="70%" alt="全体図" src="https://user-images.githubusercontent.com/13707135/86593100-99f00180-bfcf-11ea-89dc-4fe04f0ed80f.png">

まず、トップレベルタスク（大きめのタスク）をPMメンバーが担当します。  
PMはサブタスクとなるissueを作り、slackの [#proj_salonapp](https://kboy.slack.com/archives/C016RAU4H52) を通じて対応してくれるエンジニアを募集します。基本的には早い者勝ちで、手を上げてくれたエンジニアさんにアサインされます。  
エンジニアは別々のPMのサブタスクを複数掛け持ちすることも可能です。  
サブタスクが全部終わったら（＝トップレベルタスクも終わり）、PMはプロダクトオーナーに報告してレビューを受けます。  
プロダクトオーナーが確認して問題なければトップレベルタスクが完了になります。

ラフなスクラム（開発見積りとかデイリースクラムとかしないスクラム）開発に、OSSのエッセンスを加えた感じ。

## ロールとルールの定義

### プロダクトオーナー：KBOY
* プロダクトに責任を持つ
* トップレベルタスクのOpen/アサイン/Closeの権限を持つ
* マスターのアウトプットをレビューする
* トップレベルタスクを横断する問題を解決する

### PM
* トップレベルタスクに責任を持つ
* サブタスクのOpen/アサイン/Closeの権限を持つ
* アサインせずにエンジニアから担当者を募集してよい
* エンジニアのアウトプットをレビューする

### エンジニア
* サブタスクに責任を持つ
* 質問等はマスターに行う
